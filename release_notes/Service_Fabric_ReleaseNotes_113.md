## Service Fabric Feature and Bug Fixes

| **Versions** | **Issue Type** | **Description** | **Resolution** |
|---|---|---|---|
| Windows - <br> 11.3.365.1 <br> Ubuntu 22 - <br> 11.3.365.4 | Feature | Zstandard Compression | **Brief Description:** This release (11CU3) introduces support for the new Zstandard compression method, which can be configured either through BackupPolicy or Cluster configuration. Zstandard provides improved compression ratios and faster performance compared to traditional Zip compression. <br> **Feature/Bug Impact:** <br> *Improved Performance and Efficiency* - Zstandard offers better compression ratios and significantly faster compression/decompression speeds, leading to  reduced backup sizes and faster data handling. <br> *Flexible Integration* - Can be enabled via BackupPolicy or Cluster configuration, providing flexibility in how users adopt the feature. <br> *Backward Compatible* - Existing backup policies without the new compression settings will continue to function as before, ensuring smooth upgrades and compatibility. <br> **Solution/Fix:** Users can enable Zstandard compression by updating the BackupPolicy or by upgrading the Cluster configuration. No changes are required for existing setups unless Zstandard is to be used. |
| Windows - <br> 11.3.365.1 <br> Ubuntu 22 - <br> 11.3.365.4 | Feature | Replica Soft Delete | **Brief Description:** The replica soft delete feature is an enhancement to the remove replica API workflow, where we prevent users from accidentally losing data from their stateful persisted service, and allows them to recover their data. The need stems from various incidents including Azure outages caused by the misuse of this data destructive operation. <br> **Feature/Bug Impact:** This feature helps prevent accidental data loss due the data destructive remove replica operation. <br> **Solution/Fix:** Upon deleting a replica (using PowerShell APIs, UI or FabricClient APIs), instead of dropping the replica(s) and cleaning up the data from disk, we close the replica object and mark it Down. In this “stopped” state, the replica object itself on the node is closed, which prevents it from receiving any future reads/writes. But SF still tracks this replica and prevents it from getting cleaned up. If the replica removal causes the partition to go into quorum loss, manual intervention will be required to either restore the replica using a new administrative API if this was unintentional or indicate that data loss is acceptable and we really want to get rid of the replica. <br> **Breaking Change:** The feature will be released with SF11 CU3 as an opt-in feature. However, it will be enabled by default in SF12. The behavior is a breaking change as it changes how the remove replica operation works today. |
| Windows - <br> 11.3.365.1 <br> Ubuntu 22 - <br> 11.3.365.4 | Feature | Replica Soft Delete | **Brief Description:** Replicas scheduled for removal with soft-delete feature enabled and using the -ForceRemove pattern will not wait for the replica to close gracefully but will still be placed in a soft-deleted state. <br> **Feature/Bug Impact:** Replicas scheduled for removal with soft-delete feature enabled and using the -ForceRemove pattern will not wait for the replica to close gracefully but will still be placed in a soft-deleted state. This means that all current replica removals accomplished through SFE, Powershell or the FabricClient will perform the soft-delete behavior regardless if ForceRemove was set to true or not. <br> **Solution/Fix:** Helps prevent data loss scenarios due to removal of replicas using ForceRemove which contain necessary data that if lost, would result in losing data for the entire partition. <br> **Workaround:** If customers would like to remove replicas forcibly without having the soft-delete behavior, either the feature flag must be disabled or the new RemoveReplicaAsync FabricClient API that takes in RemoveReplicaOptions must be used with PurgeImmediately flag set. |
| Windows - <br> 11.3.365.1 <br> Ubuntu 22 - <br> 11.3.365.4 | Feature | Get-ServiceFabricReplica | **Brief Description:** Get-ServiceFabricReplica now includes a new possible status ToBeRemoved for replicas that are experiencing a delayed cleanup. If the status is ToBeRemoved, the command will also include information on the expiration time of the ToBeRemoved replica at which point the replica will be permenantly deleted if the partition is in a healthy state. <br> **Feature/Bug Impact:** Get-ServiceFabricReplica includes new information for replicas that are experiencing delayed cleanup including their expiration time. <br> **Solution/Fix:** Makes visible information to customers regarding a delayed cleanup replicas expiration time. |
| Windows - <br> 11.3.365.1 <br> Ubuntu 22 - <br> 11.3.365.4 | Feature | Restore Replica API | **Brief Description:** This new restore replica API is a part of the replica soft delete feature, which is an enhancement to the remove replica API workflow, where we prevent users from accidentally losing data from their stateful persisted service, and allows them to recover their data. The need stems from various incidents including Azure outages caused by the misuse of this data destructive operation. <br> **Feature/Bug Impact:** This feature helps prevent accidental data loss due the data destructive remove replica operation. <br> **Solution/Fix:** With the replica soft delete feature enabled, if the replica removal causes the partition to go into quorum loss, manual intervention will be required to either restore the replica using this new administrative API if this was unintentional or indicate that data loss is acceptable and we really want to get rid of the replica. |
| Windows - <br> 11.3.365.1 <br> Ubuntu 22 - <br> 11.3.365.4 | Feature | BRS-Enabled Partition Recovery | **Brief Description:** This feature helps a Backup Restore Service-enabled partition recover immediately from Partial Data loss if the data on Primary is more recent than the data in the last backup. <br> **Feature/Bug Impact:** <br> *Improved Availability* - Reduces the Partition Down time in case of Partial Data loss because the partition doesn't wait for user intervention to restore. <br> *Backward Compatible* - It will work fine for the partitions with policies which doesn't include this parameter as this is not a required parameter with default value disabled. <br> **Solution/Fix:** User needs to add the Quick Recovery feature in their Backup Policies to "FromPrimary" to enable this feature. By default it will be set to Disabled. |
| Windows - <br> 11.3.365.1 <br> Ubuntu 22 - <br> 11.3.365.4 | Feature | RemoveReplicaAsync | **Brief Description:** All current usage of RemoveReplicaAsync through the FabricClient APIs and by extension Powershell and SFE now will by default invoke the soft-delete behavior if the feature flag is set. A new set of APIs for RemoveReplicaAsync are introduced which taken in a RemoveReplicaOptions parameter that can allow overriding the soft-delete behavior and more clearly define the flags being set for a replica removal. Some key existing usages where replicas should be purged immediately have been updated to use the new API with the flag set. <br> **Feature/Bug Impact:** If the soft-delete feature flag is set, all existing customer defined remove replica requests (including those from Powershell and SFE) will perform the soft-delete behavior. In order to bypass this behavior if the cluster flag is set, the new API which takes in RemoveReplicaOptions must be used which is available in the 11.3 version of the Service Fabric SDK. <br> **Solution/Fix:** Helps prevent data loss scenarios due to removal of replicas using ForceRemove which contain necessary data that if lost, would result in losing data for the entire partition. <br> **Breaking Change:** Yes, if the soft-delete feature flag is set, all remove replica requests will now soft-delete the replicas which is different than the existing behavior which purges the replica immediately. <br> **Workaround:** If customers would like to remove replicas forcibly without having the soft-delete behavior, either the feature flag must be disabled or the new RemoveReplicaAsync FabricClient API that takes in RemoveReplicaOptions must be used with PurgeImmediately flag set. |
| Windows - <br> 11.3.365.1 <br> Ubuntu 22 - <br> 11.3.365.4 | Feature | SFX UI | **Brief Description:** SFX will now include a ToBeRemoved replica's expiration time in the essentials tab and display that a replica is ToBeRemoved in the left pane. <br> **Feature/Bug Impact:** SFX will now include a ToBeRemoved replica's expiration time in the essentials tab and display that a replica is ToBeRemoved in the left pane. <br> **Solution/Fix:** Makes visible information to customers regarding a delayed cleanup replicas expiration time and indicate that a replica is ToBeRemoved in the left pane. |
| Windows - <br> 11.3.365.1 <br> Ubuntu 22 - <br> 11.3.365.4 | Feature | Reliable Dictionary Health Events | **Brief Description:** Reliable Dictionary based services will now emit health events if customer defined single-item notification callbacks exceed a given threshold (defaulted to 1000ms). This will inform customers if their defined code is slowing down the reliable dictionary applies as notifications are a blocking operation defined by the customer which can result in performance impact if notifications are repeatedly long running. <br> **Feature/Bug Impact:** Observability improvement to notify customers via health events if their single-item notifications for Reliable Dictionaries exceed a configurable threshold. These health events are emitted with a TTL of 1min and can at most be emitted once every 15 minutes for a replica. <br> **Solution/Fix:** Improved observability for long running notifications in Reliable Dictionaries. <br> **Workaround:** If customers do not want to be notified of long running notifications, they can set TransactionalReplicator's NotificationCallbackThresholdInMilliseconds setting to 0 which disables the feature. |
| Windows - <br> 11.3.365.1 <br> Ubuntu 22 - <br> 11.3.365.4 | Bug | Sleep() Function | **Brief Description:** This update ensures that the Sleep() function is not called when the associated internal FM config (FMJobProcessingDelayDurationMs) is set to 0. <br> **Feature/Bug Impact:** Currently, when FM processes the message from the CommonQueue after dequeuing, the Sleep() function is called with the value of FMJobProcessingDelayDurationMs (FM internal config). This function and its associated config were introduced to simulate the common queue filled scenario for testing purposes only (FM.CommonQueueFull.test). In non-testing scenarios, the Sleep function should ideally not be called. However, it is currently called with Sleep(0) since FMJobProcessingDelayDurationMs is always 0 in non-testing scenarios. Sleep(0) can yield inconsistent behavior by potentially causing the thread to yield or sleep, depending on thread scheduling. <br> **Solution/Fix:** Sleep() is not called when the associated internal FM config (FMJobProcessingDelayDurationMs) is set to 0. |
| Windows - <br> 11.3.365.1 <br> Ubuntu 22 - <br> 11.3.365.4 | Bug | Failover Manager | **Brief Description:** This update ensures that Failover Manager (FM) continues to send RemoveInstance messages to Resource Agent (RA) even after node reactivation, if the node was previously deactivated destructively. <br> **Feature/Bug Impact:** Previously, FM stopped sending RemoveInstance calls to RA after a node reactivation. It causes the RA and FM state mismatched, for the following edge case : RA was still processing the original RemoveInstance request and became stuck in IStatelessServiceInstance.Close(). While RA remained in this stuck state, the user process terminated. However, RA did not notify FM because the instance had already been marked as deleted. Without continued RemoveInstance messages from FM, RA had no opportunity to reply and synchronize the state. The correct behavior is for FM to continue sending RemoveInstance messages even after node reactivation, ensuring that RA and FM states remain synchronized. <br> **Solution/Fix:** FM logic is updated to continue sending RemoveInstance messages post-reactivation for nodes that were destructively deactivated (Stateless Service type only). |

## Retirement and Deprecation Path Callouts

* **Important:** Certificate Authorities are removing ClientAuth EKU from publicly trusted TLS certificates between May and October 2026. Service Fabric clusters using publicly issued certificates must ensure their cluster and client certificates retain client EKU capability to avoid service disruptions. For detailed information and action items, please see the [Client EKU Removal Addendum](Resources/ClientEKURemoval.md).

* Service Fabric (SF) runtime will discontinue support for the Java SDK soon. For a smooth transition, we strongly recommend users to shift to Azure Service Fabric .NET SDK. If your current setup is based on the Service Fabric Java SDK, we suggest starting migration plans to smoothly switch to the Azure Service Fabric .NET SDK. Although applications using the Java SDK will continue to work, we highly recommend adopting the SF .NET SDK for optimal outcomes.

* Service Fabric runtime will soon be archiving and removing Service Fabric runtime versions less than 7.2 and older, as well as the corresponding SDK version 4.2 packages and older from the package Download Center. Archiving/Removing will affect application scaling and re-imaging of virtual machines in a Service Fabric Cluster running on unsupported versions. After older versions are removed/archived, this may cause failure while rolling back when the current in-progress upgrade has errors.
  * To prevent disruption of workloads, create a new cluster using the following steps:
    * [Create a Service Fabric cluster using ARM template](/azure/service-fabric/quickstart-cluster-template)
    * [Create a Standalone cluster](/azure/service-fabric/service-fabric-cluster-creation-for-windows-server)
    * Install the supported version of Service Fabric SDK based on the Runtime version installed on the cluster.

* Starting in Service Fabric 11 CU3, support for stateless instance removal will begin its deprecation pathway. Support for stateless instance removal will be fully removed approximately one year after the release of 11 CU3.
  * The following cmdlets and API calls will be deprecated for stateless replicas:
    * [Remove-ServiceFabricReplica](https://learn.microsoft.com/en-us/powershell/module/servicefabric/remove-servicefabricreplica)
    * [FabricClient.FaultManagementClient.RemoveReplicaAsync](https://learn.microsoft.com/en-us/dotnet/api/system.fabric.fabricclient.faultmanagementclient.removereplicaasync)
    * [Remove Replica – Microsoft Learn](https://learn.microsoft.com/en-us/rest/api/servicefabric/sfclient-api-removereplica)
  * Support for stateless instance removal will be fully removed approximately one year after the release of 11 CU3:
  * **Customer impact:** Using these APIs after upgrading to version 11 CU3 will trigger deprecation warnings. Full removal of support is targeted for a future release, approximately one year post-11.3.
    * Scripts and tools depending on the deprecated APIs may fail in future versions.
    * Customers should begin migrating to the new APIs as soon as they become available to prevent disruptions in automation workflows.
    * Alternate APIs to support stateless instance removal will be announced in a 11 CU3 release.
  * **Migration path:** Service Fabric 11 CU3 introduces new APIs and a compatibility switch to ensure existing scripts continue functioning after upgrading, while allowing time to transition.
    * Full details about replacements and migration guidance will be included in the 11 CU3 release notes and documentation.
